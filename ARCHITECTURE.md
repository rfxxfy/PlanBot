## 🔄 Поток данных

### 1. Добавление задачи

```
Пользователь: /addtask Название | 4 | 8 | 25.12.2025
       │
       ▼
HandleUpdate() → handleAddTask()
       │
       ├─→ Парсинг параметров
       ├─→ Валидация
       └─→ database.CreateTask()
              │
              ▼
           INSERT в БД (tasks)
              │
              ▼
        Ответ пользователю: "✅ Задача создана"
```

### 2. Планирование задач

```
Пользователь: /schedule
       │
       ▼
HandleUpdate() → handleSchedule()
       │
       ├─→ database.GetPendingTasks()
       │        │
       │        ▼
       │   SELECT * FROM tasks WHERE status='pending'
       │        │
       │        ▼
       └─→ scheduler.NewScheduler(user, tasks)
              │
              ▼
       Schedule(startDateInUserTZ) [Алгоритм]
              │
              ├─→ sortTasksByDeadlineAndPriority()
              ├─→ scheduleTask() для каждой задачи
              │      │
              │      ├─→ Проверка дедлайна
              │      ├─→ Поиск свободных слотов
              │      └─→ Распределение часов
              │
              ▼
       ScheduleResult
              │
              ├─→ database.ClearTaskSchedules()
              └─→ database.SaveTaskSchedules()
                     │
                     ▼
              INSERT в БД (task_schedules)
              UPDATE tasks SET status='scheduled'
                     │
                     ▼
              Ответ пользователю с расписанием
```

### 3. Просмотр расписания

```
Пользователь: /today или /week
       │
       ▼
HandleUpdate() → handleToday()/handleWeek()
       │
       └─→ database.GetScheduleForDateRange()
              │
              ▼
       SELECT с JOIN tasks и task_schedules
              │
              ▼
       Форматирование → DaySchedule[]
              │
              ▼
       Ответ пользователю с расписанием
```

## 🗄️ Модель данных

### Таблицы БД

```sql
users
├─ id (PK)
├─ telegram_id (UNIQUE)
├─ username
├─ first_name
├─ last_name
├─ time_zone (строка, IANA name, по умолчанию Europe/Moscow)
├─ work_start (время начала рабочего дня, HH:MM)
├─ work_end (время окончания рабочего дня, HH:MM)
├─ daily_capacity (часов в день, суммарная ёмкость)
├─ work_days (массив рабочих дней недели)
├─ created_at
└─ updated_at

tasks
├─ id (PK)
├─ user_id (FK → users.id)
├─ title
├─ description
├─ hours_required
├─ priority (1-10)
├─ status (pending/scheduled/completed...)
├─ deadline
├─ created_at
├─ updated_at
└─ completed_at

task_schedules
├─ id (PK)
├─ task_id (FK → tasks.id)
├─ scheduled_date
├─ hours_allocated
└─ created_at
```

### Связи

```
users (1) ──< (N) tasks
tasks (1) ──< (N) task_schedules
```

## 🧩 Компоненты системы

### 1. main.go
**Назначение**: Точка входа приложения

**Ответственность**:
- Загрузка конфигурации (.env)
- Инициализация подключения к БД
- Создание экземпляра бота
- Запуск обработчика обновлений

### 2. handlers/handlers.go
**Назначение**: Обработка команд бота

**Ответственность**:
- Роутинг команд (/start, /addtask, /schedule...)
- Парсинг аргументов команд
- Валидация пользовательского ввода
- Форматирование ответов для Telegram
- Взаимодействие с БД и планировщиком

**Основные функции**:
- `HandleUpdate()` - главный обработчик
- `handleStart()` - приветствие
- `handleAddTask()` - добавление задачи
- `handleSchedule()` - запуск планирования
- `handleToday()/handleWeek()` - просмотр расписания
- `handleSettings()` - изменение дневной ёмкости, рабочих дней и рабочего времени
- `handleTimezone()` - установка таймзоны пользователя

### 3. scheduler/scheduler.go
**Назначение**: Алгоритм планирования задач

**Ответственность**:
- Сортировка задач по приоритету и дедлайнам
- Распределение задач по дням
- Учёт дедлайнов и рабочих дней
- Разбиение задач на несколько дней
- Учет горизонта планирования (`PLANNING_HORIZON_DAYS`)

**Основные функции**:
- `Schedule(startDate time.Time)` - главная функция планирования (стартовая дата уже приведена к таймзоне пользователя)
- `sortTasksByDeadlineAndPriority()` - сортировка задач с дедлайнами и без
- `scheduleTask()` - распределение одной задачи по дням с учетом рабочих дней и дедлайна
- `calculateLatestStartDate()` - расчет последней допустимой даты начала с использованием `ceil(hours_required / daily_capacity)`

**Алгоритм**: O(n × d), где n - задач, d - дней

### 4. database/queries.go
**Назначение**: Работа с базой данных

**Ответственность**:
- CRUD операции для users/tasks/task_schedules
- Транзакции для согласованности данных
- Преобразование между Go типами и SQL

**Основные функции**:
- `GetOrCreateUser()` - получение/создание пользователя
- `CreateTask()` - создание задачи
- `GetPendingTasks()` - получение незапланированных задач
- `SaveTaskSchedules()` - сохранение расписания
- `GetScheduleForDateRange()` - получение расписания

### 5. database/db.go
**Назначение**: Управление подключением к БД

**Ответственность**:
- Создание connection pool
- Проверка соединения
- Graceful shutdown

### 6. models/models.go
**Назначение**: Определение структур данных

**Структуры**:
- `User` - пользователь с настройками
- `Task` - задача с параметрами
- `TaskSchedule` - запись расписания
- `DaySchedule` - расписание на день
- `ScheduleResult` - результат планирования

## 🎯 Принципы проектирования

### 1. Разделение ответственности (SoC)
- Каждый пакет отвечает за свою область
- handlers - UI логика
- scheduler - бизнес-логика планирования
- database - персистентность

### 2. Зависимости
```
main → handlers → scheduler → models
   ↓           ↓
database ← ─ ┘
```

### 3. Обработка ошибок
- Ошибки БД логируются и возвращаются
- Пользователю показываются понятные сообщения
- Нет паники в runtime

### 4. Конфигурация
- Все настройки через переменные окружения
- Использование .env файла для локальной разработки
- Безопасное хранение секретов

## 🔐 Безопасность

### 1. База данных
- Использование подготовленных запросов (prepared statements)
- Защита от SQL инъекций
- Каскадное удаление связанных данных

### 2. Telegram Bot
- Токен хранится в переменных окружения
- Валидация пользовательского ввода
- Ограничение размера данных

### 3. Бизнес-логика
- Проверка прав доступа (user_id)
- Валидация дат и числовых значений
- Обработка edge cases

## 📈 Масштабируемость

### Текущие ограничения:
- Один инстанс бота
- Один connection pool к БД
- Синхронная обработка сообщений

### Возможности для масштабирования:

1. **Горизонтальное масштабирование**
   - Использование webhook вместо long polling
   - Load balancer перед несколькими инстансами
   - Распределенное хранилище сессий

2. **Оптимизация БД**
   - Индексы на часто запрашиваемые поля ✅ (уже есть)
   - Партиционирование таблицы task_schedules по датам
   - Read replicas для чтения

3. **Кэширование**
   - Redis для кэширования расписаний
   - In-memory кэш пользовательских настроек

4. **Асинхронность**
   - Очередь задач для тяжелых операций
   - Background workers для планирования
