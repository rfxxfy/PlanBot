# План доработки PlanBot

## Обзор текущего состояния

- **Есть рабочий MVP**:
  - Телеграм‑бот с командами `/start`, `/help`, `/addtask`, `/mytasks`, `/schedule`, `/today`, `/week`, `/complete`, `/delete`, `/settings` (`handlers/handlers.go`).
  - Алгоритм планирования по дням с дедлайнами и приоритетами (`scheduler/scheduler.go`).
  - PostgreSQL‑схема и CRUD‑слой (`database/schema.sql`, `database/queries.go`, `database/db.go`).
  - Доменные модели (`models/models.go`), health‑сервер (`health/health.go`), Docker/Compose, Makefile, CI/CD.
- **Не реализовано из большого ТЗ (`TASK.md`)**:
  - Таймзоны пользователей, рабочие часы (start/end).
  - Планирование по временным слотам внутри дня.
  - Интеграция с Google Calendar.
  - Расширенный UX (диалоговый ввод задач, inline‑кнопки).
  - Мониторинг (Prometheus/Grafana) и часть прод‑инфраструктуры.

## Этап 1. Дошлифовать текущий MVP (без тестов)

  - Заменить приближение `int(hoursRequired/DailyCapacity) + 1` на расчёт через `ceil(hoursRequired / DailyCapacity)`.
  - Явно зафиксировать в комментарии допущения относительно рабочих дней.
- **1.1.3**: Добавить более строгий режим «обратного» планирования от дедлайна:
  - Для задач с дедлайном сначала пытаться заполнять дни, максимально близкие к дедлайну, в пределах допустимого окна.
  - Сохранить текущий прямой режим от `startDate` как fallback.
- **1.1.4**: Ввести ограничение горизонта планирования:
  - Ввести конфигурационный параметр (например, `PLANNING_HORIZON_DAYS`, по умолчанию 90).
  - В `Schedule`/`scheduleTask` прекращать просмотр будущих дней по достижении лимита и относить такие задачи к `UnscheduledTasks`.

### 1.2. Улучшение UX команд бота

- **1.2.1**: Прочистить и унифицировать тексты ответов в `handlers/handlers.go`:
  - Сделать формулировки ошибок более конкретными (различать формат даты, число часов, неправильный ID и т.д.).
  - Убедиться, что каждая команда всегда отвечает пользователю (нет тихих провалов).
- **1.2.2**: Расширить ответ `/schedule`:
  - При частичном успехе явно писать, что часть задач не удалось запланировать и почему (недостаток времени/жёсткие дедлайны).
  - При полном успехе добавлять резюме: сколько задач и на сколько дней запланировано.
- **1.2.3**: Доработать ответы `/today` и `/week`:
  - Если дневная нагрузка превышает `DailyCapacity`, выводить предупреждение («День перегружен (X > Y часов)»).
  - Если на период нет задач — предлагать вызвать `/schedule` или `/addtask`.
- **1.2.4**: Обновить текст `/help`:
  - Привести примеры команд в строгое соответствие с фактическим форматом парсинга.
  - Не упоминать фичи, которых нет в текущем коде (Google Calendar, временные слоты и т.п.).

### 1.3. Небольшие инфраструктурные правки

## Этап 3. Переход к планированию по временным слотам (дорожная карта)

*(На этом этапе мы проектируем модель и API, не меняя пока текущий алгоритм по дням.)*

### 3.1. Модель временных слотов

- **3.1.1**: Определить базовую доменную модель `TimeSlot` (только в `models.go`, без жёсткой привязки к БД):
  - Поля: `UserID`, `Date`, `StartTime`, `EndTime`, `CapacityHours`, `AllocatedHours`, `Source` (например: `task`, `external`).
  - Размер слота: зафиксировать первоначально 60 минут как дефолт (с возможностью конфигурации в будущем).
- **3.1.2**: Определить вспомогательную сущность/тип для дневного окна:
  - Например, `DayTimeWindow` (или использование `User.WorkStart/WorkEnd` напрямую) для генерации слотов в рамках одного дня.
- **3.1.3**: Решить стратегию хранения:
  - На первом этапе слоты генерируются **in-memory** на основе:
    - `User.WorkStart/WorkEnd`, `User.WorkDays`, `User.TimeZone`.
    - Горизонта планирования (`PLANNING_HORIZON_DAYS`).
  - В БД по‑прежнему храним только агрегированную информацию по дням (`task_schedules`), чтобы не взрывать объём данных.

### 3.2. Новый API планировщика (на уровне интерфейсов)

- **3.2.1**: Спроектировать интерфейс/контракт для планировщика слотов:
  - Вход: список задач, настройки пользователя (включая `TimeZone`, `WorkStart/End`), внешние занятости (например, события из календаря, пока как абстракцию).
  - Выход: коллекция `TimeSlot` с привязкой задач к конкретным временным окнам.
- **3.2.2**: Добавить в `scheduler` черновой скелет нового планировщика:
  - Новая структура (например, `SlotScheduler`) и методы:
    - `BuildDailySlots(user, startDate, horizon) []TimeSlot` — генерация свободных слотов по дням.
    - `AssignTasksToSlots(tasks, slots) []TimeSlot` — жадное распределение задач по слотам.
  - Пока эти методы могут быть пустыми или использующими только часть логики, без интеграции в текущий `/schedule`.

### 3.3. Подготовка БД и совместимость

- **3.3.1**: Оставить текущую схему `task_schedules` без изменений:
  - Продолжать хранить агрегированную информацию по дням (`DATE + hours_allocated`).
  - Слоты использовать как внутреннее представление, которое в конце «сворачивается» в почасовую нагрузку по дням.
- **3.3.2**: В документации (`ARCHITECTURE.md`, `ALGORITHM.md`) описать:
  - Как из набора `TimeSlot` получается текущая дневная модель (`DaySchedule` и `task_schedules`).
  - Что добавление точного времени начала/конца в БД будет отдельным шагом и не ломает совместимость с уже существующими данными.
